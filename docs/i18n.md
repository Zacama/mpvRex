# Internationalization (i18n)

This document describes how localization works in mpvRex, how to add a new
language, the conventions to follow, and the known limitations of the current
implementation.

## Overview

| Locale folder | Language |
|---|---|
| `app/src/main/res/values/` | English (US) — default / source of truth |
| `app/src/main/res/values-es/` | Spanish |
| `app/src/main/res/values-zh-rCN/` | Simplified Chinese |
| `app/src/main/res/values-zh-rTW/` | Traditional Chinese |

- `generateLocaleConfig = true` in `app/build.gradle.kts` auto-generates the
  Android 13+ per-app language list from the `values-*` folders;
  `app/src/main/res/resources.properties` declares the default locale
  (`unqualifiedResLocale=en-US`). No manual locale-config registration is
  needed when adding a language.
- In-app language switching lives in the Appearance settings
  (`ui/preferences/AppearancePreferencesScreen.kt`). It is implemented with
  `AppCompatDelegate.setApplicationLocales()` plus the
  `AppLocalesMetadataHolderService` manifest entry (`autoStoreLocales`), which
  persists the choice on Android 12 and below. All activities must extend
  `AppCompatActivity` for the pre-Android-13 backport to work.
- The selectable languages are listed in the `AppLanguage` enum at the bottom
  of `AppearancePreferencesScreen.kt`. **This enum is not derived from the
  resource folders** — see the checklist below.

## Adding a new language

1. Create `app/src/main/res/values-<qualifier>/strings.xml` and translate the
   entries from `values/strings.xml`. Skip entries marked
   `translatable="false"` (do not copy them into the translation).
2. Add an entry to the `AppLanguage` enum in
   `ui/preferences/AppearancePreferencesScreen.kt` with the BCP-47 tag and the
   language's name written in itself (native names are intentionally never
   translated). Extend `AppLanguage.fromTag()` if the tag needs special
   matching (as Chinese does for Hans/Hant variants).
3. Verify key consistency — extra keys in a translation fail the build, and
   placeholder mismatches crash at runtime:

   ```bash
   python3 - <<'EOF'
   import re
   def parse(p):
       src = open(p, encoding="utf-8").read()
       s = dict(re.findall(r'<string name="([^"]+)"[^>]*>(.*?)</string>', src, re.S))
       nt = set(re.findall(r'<string name="([^"]+)"[^>]*translatable="false"', src))
       return s, nt
   base, base_nt = parse("app/src/main/res/values/strings.xml")
   tr, _ = parse("app/src/main/res/values-zh-rCN/strings.xml")  # adjust folder
   print("extra keys:", sorted(set(tr) - set(base)) or "none")
   print("untranslated:", sorted(set(base) - set(tr) - base_nt) or "none")
   fmt = lambda s: sorted(re.findall(r'%(?:\d+\$)?[sdf]', s))
   print("placeholder mismatches:",
         [k for k, v in tr.items() if k in base and fmt(base[k]) != fmt(v)] or "none")
   EOF
   ```

4. Build: `./gradlew :app:assembleDebug` (resource linking validates escaping),
   then switch to the language in Appearance settings and spot-check.

Missing translations fall back to English at runtime and do not block release
builds (`lintVitalRelease` passes); only *extra* keys break the build.

## Conventions

- **Key naming**: prefix by screen/feature — `pref_<screen>_*` for settings,
  `sort_dialog_*`, `playlist_*`, `network_*`, `shorts_*`, `custom_button_*`,
  etc. Reusable single words go under `generic_*` (newer) or live as bare keys
  (`done`, `back`, `close` — historical).
- **Placeholders**: always positional (`%1$s`, `%2$d`). Translations must keep
  the same placeholder set; order may change.
- **Plurals**: use `<plurals>` for count-dependent text. Chinese (and other
  no-plural languages) only need the `other` quantity.
- **Resolution patterns**:
  - Composables: `stringResource(R.string.x)` directly.
  - Non-composable callbacks inside composables: resolve with
    `stringResource` in the composable scope first, capture the value.
  - Coroutines/toasts with an Activity context: `context.getString(...)` at
    call time.
  - **Application/service contexts** (ViewModels, services, managers): use
    `context.withAppLocale().getString(...)` (`utils/LocaleUtils.kt`). The
    AppCompat per-app-locale backport only localizes Activity contexts on
    Android 8–12, so a bare `applicationContext.getString()` would resolve
    against the system language there.

## Known limitations & tech debt

1. **SortDialog uses localized labels as identity.** The generic `SortDialog`
   (`ui/browser/dialogs/SortDialog.kt`) passes sort types around as localized
   strings and maps them back to enums by label comparison. Consequence: the
   translations of *Title / Duration / Date / Size / Count* must be distinct
   within each language. A collision degrades safely (the tap is ignored, no
   crash) but breaks sorting selection. Proper fix if this ever bites:
   make `SortDialog` carry typed values (e.g. `List<Pair<T, String>>`)
   instead of bare strings.
2. **`AppLanguage` must be kept in sync manually** with the `values-*`
   folders (see checklist above). The system-settings language list is
   generated automatically; the in-app picker is not.
3. **`rename_dialog_title` is a compound message** ("Rename %1$s" + a noun
   from `item_type_*`). Fine for English/Spanish/Chinese; languages with
   grammatical case or gender would need per-type titles instead.
4. **`values-es` and `values-zh-rTW` are missing ~350 newer keys** (everything
   added when hardcoded strings were externalized). They fall back to English
   at runtime. Backfilling them is pending work.
5. **Notification channel names are fixed at first creation** — Android does
   not re-read the channel name when the app language changes later
   (platform limitation, not specific to this app).
